⚙️ Required Datastores

These core internal infrastructure components are bootstrapped sequentially via the `docker-compose` topology before any microservice begins executing JVM profiles.

  • PostgreSQL :5432 Used strictly for generic physical persistence.
  • RabbitMQ :5672 Broker protocol routing message queues between disconnected nodes.
  • Redis :6379 LRU volatile caching and real-time pub/sub features.

🔍 Infrastructure Deep-Dive

☁️ AWS EC2 Instance

The entire microservices topology, datastores, and message brokers are orchestrated onto a single high-performance Linux EC2 machine (e.g., t3.xlarge) inside a private VPC, protected strictly via AWS Security Groups.

  • > SG Rule: ALLOW Inbound 443 (HTTPS)
  • > SG Rule: ALLOW Inbound 80 (HTTP) -> Redirect 443
  • > ALL other internal ports (8081, 5432, 6379) BLOCKED externally.

🐳 Docker Orchestration

Containers are bound strictly to a custom Docker bridge network. Service resolution happens via internal Docker DNS, meaning the Gateway routes to http://user-service:8082 without needing physical IP lookups.

  • > Healthchecks implemented for boot sequencing.
  • > Restart: always policy ensures immediate crash recovery.

🛡️ Nginx Deep Proxy

Nginx runs natively on the EC2 host (outside Docker) acting as the ultimate boundary. It implements GZIP compression, limits payload sizes to prevent DDoS bursts, and proxies specific path prefixes to the Internal Gateway.

  • > ProxyPass /api/ -> http://127.0.0.1:8080/
  • > Rate_limit: 10r/s per user IP zone.

📊 Metrics & Monitoring

Each Spring Boot application exposes standard /actuator/prometheus endpoints. A localized Prometheus time-series database scrapes memory, GC metrics, and HTTP latency intervals every 15s.

  • > Grafana Dashboard securely mounted visualizing TPS.
  • > Alerts configured on >90% JVM Heap limits.
