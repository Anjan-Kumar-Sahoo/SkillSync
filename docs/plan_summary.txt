SkillSync Project - Backend Implementation Summary
==================================================

1. OTP Email Authentication (Implemented in Auth-Service)
---------------------------------------------------------
- Objective: Ensure email verification before full access.
- Mechanism:
    * User registration triggers an OTP (6-digits).
    * Sent via EmailService using JavaMailSender (from: aksahoo1097@gmail.com).
    * OTP stored in 'otp_tokens' table with 5-minute expiry.
    * Endpoints: POST /api/auth/verify-otp, POST /api/auth/resend-otp.
    * Security: Retry limit (5 attempts), Async sending, Scheduled cleanup of expired tokens.

2. Entity Design & Best Practices
---------------------------------
- Type Choice (Long vs long):
    * We use the 'Long' wrapper class for IDs and foreign keys across all services.
    * Reason: 'Long' supports NULL values. This is critical in JPA because an entity's ID is NULL before it is saved to the database. Primitives (long) defaults to 0, which can lead to accidental overwrites or logical errors during persistence.
- Identity: All entities use IDENTITY generation strategy for PostgreSQL compatibility.

3. Data Handling Patterns
-------------------------
- Records (Java 16+):
    * Used for all DTOs (e.g., RegisterRequest, AuthResponse).
    * Benefit: Immutable, thread-safe, and removes boilerplate code (no need for getters, toString, or equals).
- Paging:
    * Implemented in all listing services (User, Skill, Mentor, Session, Review, Notification).
    * Uses Spring Data Pageable interface.
- Indexing:
    * Databases (PostgreSQL) automatically index Primary Keys and Unique columns.
    * Explicit indexing can be added to search columns (like skill name) if performance lags.
- Caching:
    * Currently using Database Persistence for reliability.
    * Redis can be integrated as a "DB Cache" for hot data (like active sessions or frequently searched skills) to reduce DB load.

4. Service Architecture Summary
-------------------------------
- Eureka: Discovery Service.
- Config-Server: Centralized properties.
- API-Gateway: Routing and Authentication filter.
- Auth-Service: JWT Management + OTP Verification.
- User-Service: Profile, Mentor, Group, and Payment (Razorpay) domains.
- Skill/Session/Notification Services: Domain logic and micro-storage.

5. Payment Gateway (Razorpay)
-----------------------------
- Objective: Handle mentor onboarding fees and session booking fees easily.
- Mechanism:
    * Implemented entirely within User-Service (no separate DB/service).
    * Order creation via Razorpay API -> signature verification upon completion.
    * Webhook simulation for post-payment state transitions (e.g., approve mentor profile).
